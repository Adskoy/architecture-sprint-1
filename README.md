# Спринт 1: Микрофронтенды и разбивка монолитной системы на микросервисы

## Декаплинг монолитного фронтэнда Mesto

> 💻 Проект запускается через `docker compose up` из корня, хост будет доступен по адресу http://localhost:8000

### Как и при помощи чего нарезаем

Отбросив факт нецелесообразности разделения настолько небольшого проекта, мы можем разнести его на несколько 
микрофронтэндов. Для реализации этого дела был выбран Webpack Module Federation. На то есть ряд причин:

- Нет требования использовать несколько клиентских фреймворков;
- Есть небольшая потребность шарить код между модулями;
- Клиентская композиция и runtime-интеграция
- Исходное приложение и так собирается webpack'ом, а значит мы не будем тянуть лишнюю зависимость.

По DDD у нас получаются следующие микрофронтэнды со следующим функционалом:

- `auth`: регистрация/авторизация;
- `profile`: редактирование профиля;
- `cards`: управление фотографиями и лайками;
- `shared`: проект для повторного использования кода в разных микрофронтэндах, бандлится при сборке ибо не должен часто изменяться.

При этом оставшийся обрубок в директории `/frontend` выступает в роли композитора, собирающего компоненты по 
микрофронтэндам. В нем остаются компоненты оболочки (футер с хэдером).

На текущем уровне разделения части приложения у нас между собой не общаются 
(за исключением проброса пользователя в контекст), поэтому взаимодействие микрофронтэндов идёт на основе API.

### Дальнейшие шаги по оптимизации

Нет предела совершенству, и этот проект не исключение. Вот что можно (а скорее нужно) сделать далее:

- Распотрошить `App.js` и `Main.js` хоста отдельные компоненты. 
Сейчас они просто импортируют карточки и окна с микрофронта, но лучше сделать условную страницу фотографий
в микрофронте `cards`, которая будет заниматься загрузкой списка мест и его отрисовкой.
По аналогии баннер с профилем пользователя можно вынести отдельным компонентом и запихать в `profile`;
- Вынести лайки на фотографиях в отдельный микрофронтэнд. На текущем этапе это будет избыточно,
но представив стратегию развития сервиса можно предположить, что в дальнейшем у нас появится и другая статистика 
(число просмотров и показов в ленте, например);
- Настроить глобальный стейт. Сейчас это, опять же, избыточно - но в будущем на фотографиях 
может появиться попап со списком имен и аватарок пользователей их лайкнувших. 
В таком случае после обновления своего профиля мы будем продолжать лицезреть устаревшие имя/аватар.
Подойдет простенькая либа типа [recoil](https://recoiljs.org/) - мелкая, функциональная и простая как джве копейки;
- Вынести микрофронтэнды в отдельные репозитории;
- Сделать нормальные докерфайлы/компоуз.
